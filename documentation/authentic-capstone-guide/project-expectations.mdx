# Project Expectations

A Capstone project should be:

* A real problem
* Large enough for **6-7 students**
* Technically non-trivial
* Collaborative across multiple workstreams
* Feasible in one semester
* Suitable for iterative demos

If 1-2 students could finish it comfortably, it is probably too small.

## Good signs

* Multiple subsystems or integrations
* Clear users and stakeholders
* Real engineering tradeoffs
* Room for frontend, backend, testing, infrastructure, or UX work

<details>
<summary>Examples of complexity that usually fit Capstone</summary>

* API integrations
* Real-time or multi-user workflows
* AI, analytics, or data processing
* Security or authentication
* Hardware, IoT, or embedded integration
* Deployment or operational constraints

</details>

<details>
<summary>Common reasons proposals are too weak</summary>

* Mostly a static website
* Basic CRUD with little complexity
* No real stakeholder need
* No meaningful team interdependence
* Scope that is vague or unrealistically large

</details>
